# 网络请求机制问题分析

> 本文档分析阅读App网络请求机制中存在的潜在问题和改进建议。

---

## 一、安全性问题

### 1.1 SSL证书验证被绕过

**问题位置：**
- [SSLHelper.kt](/app/src/main/java/io/legado/app/help/http/SSLHelper.kt)
- [BackstageWebView.kt:222-228](/app/src/main/java/io/legado/app/help/http/BackstageWebView.kt#L222-L228)

**问题描述：**

```kotlin
// SSLHelper.kt - 信任所有证书
val unsafeTrustManager: X509TrustManager = object : X509TrustManager {
    override fun checkClientTrusted(...) {}
    override fun checkServerTrusted(...) {}  // 不做任何验证
    override fun getAcceptedIssuers() = arrayOf()
}

// BackstageWebView.kt - WebView中直接忽略SSL错误
override fun onReceivedSslError(view, handler, error) {
    handler?.proceed()  // 无条件继续
}
```

**风险：**
- 中间人攻击（MITM）风险
- 用户数据可能被截获
- 无法验证服务器身份

**改进建议：**
1. 默认关闭`unsafeSsl`选项
2. WebView的SSL错误应该根据`AppConfig.unsafeSsl`决定是否继续
3. 提供证书锁定（Certificate Pinning）选项

### 1.2 敏感信息日志泄露

**问题位置：** [UrlRecordInterceptor.kt](/app/src/main/java/io/legado/app/help/http/UrlRecordInterceptor.kt)

**问题描述：**

```kotlin
// 虽然有URL脱敏，但请求头和Cookie可能包含敏感信息
val requestHeaders = request.headers.toMap()
val userAgent = request.header("User-Agent")
val cookies = request.header("Cookie")

DebugEventCenter.emit(DebugEvent(
    ...
    requestHeaders = requestHeaders,  // 可能包含Authorization等敏感头
    cookies = cookies,                 // Cookie敏感信息
    ...
))
```

**风险：**
- 调试日志可能泄露认证Token
- Cookie信息可能被记录到数据库

**改进建议：**
1. 对敏感请求头（Authorization、Cookie等）进行脱敏
2. 提供日志级别控制
3. 敏感数据不写入持久化存储

## 二、资源管理问题

### 2.1 代理客户端缓存无限制

**问题位置：** [HttpHelper.kt:25-27](/app/src/main/java/io/legado/app/help/http/HttpHelper.kt#L25-L27)

**问题描述：**

```kotlin
private val proxyClientCache: ConcurrentHashMap<String, OkHttpClient> by lazy {
    ConcurrentHashMap()  // 没有大小限制
}

fun getProxyClient(proxy: String?): OkHttpClient {
    ...
    proxyClientCache[proxy] = proxyClient  // 无限添加
    return proxyClient
}
```

**风险：**
- 如果用户频繁更换代理配置，缓存会无限增长
- 每个OkHttpClient实例占用较多内存
- 可能导致OOM

**改进建议：**
```kotlin
// 使用LRU缓存限制大小
private val proxyClientCache = object : LruCache<String, OkHttpClient>(10) {
    override fun sizeOf(key: String, value: OkHttpClient): Int = 1
}
```

### 2.2 书源信号量映射无清理

**问题位置：** [MainViewModel.kt:60](/app/src/main/java/io/legado/app/ui/main/MainViewModel.kt#L60)

**问题描述：**

```kotlin
private val sourceSemaphores = ConcurrentHashMap<String, Semaphore>()

// 只添加，从不清理
val semaphore = sourceSemaphores.getOrPut(source.bookSourceUrl) { Semaphore(1) }
```

**风险：**
- 用户删除书源后，Semaphore不会被清理
- 长期使用后map会越来越大

**改进建议：**
1. 在书源删除时清理对应的Semaphore
2. 使用WeakHashMap或定期清理
3. 添加最大容量限制

### 2.3 UrlRecordInterceptor协程作用域泄漏

**问题位置：** [UrlRecordInterceptor.kt:35](/app/src/main/java/io/legado/app/help/http/UrlRecordInterceptor.kt#L35)

**问题描述：**

```kotlin
object UrlRecordInterceptor : Interceptor {
    private val scope = CoroutineScope(SupervisorJob() + Dispatchers.IO)
    
    fun cancelAll() {
        scope.cancel()  // 只能取消一次，之后无法恢复
    }
}
```

**风险：**
- `cancelAll()`调用后scope永久失效
- 没有生命周期感知
- 可能导致协程泄漏

**改进建议：**
```kotlin
// 使用生命周期感知的作用域
// 或者提供重置机制
fun resetScope() {
    scope.cancel()
    scope = CoroutineScope(SupervisorJob() + Dispatchers.IO)
}
```

## 三、并发控制问题

### 3.1 并发限制器无限等待

**问题位置：** [ConcurrentRateLimiter.kt:103-111](/app/src/main/java/io/legado/app/help/ConcurrentRateLimiter.kt#L103-L111)

**问题描述：**

```kotlin
suspend fun getConcurrentRecord(): ConcurrentRecord? {
    while (true) {  // 无限循环
        try {
            return fetchStart()
        } catch (e: ConcurrentException) {
            delay(e.waitTime)  // 可能等待很长时间
        }
    }
}
```

**风险：**
- 如果服务器持续限流，协程会无限等待
- 没有超时机制
- 可能导致用户操作无响应

**改进建议：**
```kotlin
suspend fun getConcurrentRecord(
    maxWaitTime: Long = 30_000L  // 最大等待30秒
): ConcurrentRecord? {
    val startTime = System.currentTimeMillis()
    while (true) {
        if (System.currentTimeMillis() - startTime > maxWaitTime) {
            throw TimeoutCancellationException("并发等待超时")
        }
        try {
            return fetchStart()
        } catch (e: ConcurrentException) {
            delay(min(e.waitTime, maxWaitTime - (System.currentTimeMillis() - startTime)))
        }
    }
}
```

### 3.2 并发记录映射无清理

**问题位置：** [ConcurrentRateLimiter.kt:12](/app/src/main/java/io/legado/app/help/ConcurrentRateLimiter.kt#L12)

```kotlin
companion object {
```

**风险：**
- 删除书源后记录不会被清理

```kotlin
    concurrentRecordMap.entries.removeIf { 
        now - it.value.time > maxInactiveTime 
    }
}
```

## 四、Cookie管理问题

### 4.1 Cookie长度限制随机删除

**问题位置：** [CookieStore.kt:78-84](/app/src/main/java/io/legado/app/help/http/CookieStore.kt#L78-L84)

**问题描述：**

```kotlin
var ck = mapToCookie(cookieMap) ?: ""
while (ck.length > 4096) {
    val removeKey = cookieMap.keys.random()  // 随机删除
    CookieManager.removeCookie(url, removeKey)
    cookieMap.remove(removeKey)
    ck = mapToCookie(cookieMap) ?: ""
}
```

**风险：**
- 随机删除可能删除重要的Cookie（如session、token）
- 可能导致登录状态丢失
- 用户可能需要重新登录

**改进建议：**
```kotlin
// 按优先级删除：先删除非关键Cookie
val importantKeys = setOf("session", "token", "auth", "login")
while (ck.length > 4096 && cookieMap.isNotEmpty()) {
    val removeKey = cookieMap.keys
        .filter { it !in importantKeys }
        .firstOrNull() ?: cookieMap.keys.first()
    cookieMap.remove(removeKey)
    ck = mapToCookie(cookieMap) ?: ""
}
```

### 4.2 会话Cookie重启丢失

**问题描述：**

会话Cookie存储在内存中（`CacheManager.putMemory`），应用重启后会丢失。

**风险：**
- 用户重启应用后需要重新登录
- 短期会话状态无法保持

**改进建议：**
1. 提供选项将会话Cookie持久化
2. 或者使用SharedPreferences存储会话Cookie

## 五、WebView请求问题

### 5.1 JS执行重试次数过多

**问题位置：** [BackstageWebView.kt:264-278](/app/src/main/java/io/legado/app/help/http/BackstageWebView.kt#L264-L278)

**问题描述：**

```kotlin
if (retry > 30) {  // 最多重试30次
    callback?.onError(NoStackTraceException("js执行超时"))
    ...
}

val intervals = listOf(200L, 400L, 600L, 800L, 1000L)
val nextDelay = if (retry < intervals.size) intervals[retry] else intervals.last()
retry++
mHandler.postDelayed(this, nextDelay)  // 每次延迟递增
```

**风险：**
- 重试30次，每次最多延迟1秒，可能超过总超时时间
- 与`withTimeout(60000L)`冲突
- 可能导致超时后还在重试

**改进建议：**
```kotlin
// 根据总超时时间动态计算最大重试次数
private fun getMaxRetry(timeout: Long, delayTime: Long): Int {
    val avgInterval = 600L  // 平均间隔
    return ((timeout - delayTime) / avgInterval).toInt().coerceIn(5, 30)
}
```

## 六、错误处理问题

### 6.1 缺少统一重试策略

**问题描述：**

当前的重试机制分散在多处：
- `OkHttpClient.retryOnConnectionFailure(true)` - 只重试连接失败
- `UrlOption.retry` - 手动配置重试次数
- WebView的JS执行重试

**风险：**
- 不同场景重试行为不一致
- 某些临时错误没有重试
- 可能导致不必要的失败

**改进建议：**
```kotlin
// 统一的重试策略
data class RetryPolicy(
    val maxRetries: Int = 3,
    val retryDelay: Long = 1000L,
    val retryOn: Set<Int> = setOf(408, 429, 500, 502, 503, 504),
    val exponentialBackoff: Boolean = true
)

fun executeWithRetry(policy: RetryPolicy, block: suspend () -> Response): Response {
    repeat(policy.maxRetries) { attempt ->
        try {
            return block()
        } catch (e: IOException) {
            if (attempt == policy.maxRetries - 1) throw e
            val delay = if (policy.exponentialBackoff) {
                policy.retryDelay * (1 shl attempt)
            } else policy.retryDelay
            delay(delay)
        }
    }
}
```

## 七、性能问题

### 7.1 没有配置连接池参数

**问题描述：**

OkHttpClient使用默认连接池配置：
- 最大空闲连接数：5
- 空闲连接保活时间：5分钟

**风险：**
- 高并发场景可能连接不足
- 频繁创建新连接影响性能

**改进建议：**
```kotlin
.connectionPool(ConnectionPool(
    maxIdleConnections = 10,  // 增加空闲连接数
    keepAliveDuration = 5, 
    TimeUnit.MINUTES
))
```

---

### 7.2 没有限制最大并发请求数

**问题描述：**

OkHttpClient默认配置：
- `maxRequests = 64`
- `maxRequestsPerHost = 5`

**风险：**
- 可能对同一服务器发起过多请求
- 可能被服务器限流或封禁

**改进建议：**
```kotlin
.dispatcher(Dispatcher().apply {
    maxRequests = 32
    maxRequestsPerHost = 3
})
```

## 八、代码质量问题

### 8.1 异常处理不完善

**问题位置：** [OkHttpExceptionInterceptor.kt](/app/src/main/java/io/legado/app/help/http/OkHttpExceptionInterceptor.kt)

**问题描述：**

```kotlin
override fun intercept(chain: Interceptor.Chain): Response {
    try {
        return chain.proceed(chain.request())
    } catch (e: IOException) {
        throw e
    } catch (e: Throwable) {
        throw IOException(e)  // 丢失原始异常堆栈
    }
}
```

**改进建议：**
```kotlin
catch (e: Throwable) {
    throw IOException(e).apply {
        addSuppressed(e)  // 保留原始异常
    }
}
```

**改进建议：**
1. 提取为配置常量
2. 提供用户可配置选项
3. 添加配置说明文档
